LamaSync - Linux-filesync solution to avoid Syncthing

This filesync solution is just on a theory stage, a need to do some research on backends needs doing.
But the idea is to have my LamaFiles folder, and any other folder I want, easily synced between computers.
This is much of what SyncThing does - But working in repo folders, p2p with several hosts really messes
with the sync itself. And after some git-work and time, directories fall out of sync.

My wish is to have theese synced folders easily managed, set up and synced. A TUI using the OpenTUI framework
will be used for management. The entire data pipeline will take place inside my tailnet, so security isn't a 
big issue here, though it needs to be considered in some basic form.

## Backup & Sync solution
The solution would preferably do both the jobs of keeping folders mounted/synced as well as some basic folder backup.
My backup needs are really basic, the basics from my home directory, and some appdata folders in /opt. I'd like theese
folders backup up, compressed and sent to the primary file host (this will be my TrueNas).

## SeaFile
SeaFile is the only self-hosted filehosting that didn't absolutely shit itself uploading a big repo. It still "conks out"
every now and then needing a restart of either client or server. Bit it got both sync and drive (mount folder) clients
that could be very useful for a simple setup.
The hiccups and extra layer is somewhat bothering me. The SeaFile uses a 3rd party hosted S3 backend, so storage is
both safer and more predictable than any other backend I can host myself.

## RClone / Rsync
This is my consideration of "just works". But I would need something abit more - But using rclone opens things like
caching, simple mount operation, the other way is rsync to keep a folder in sync. This turns into a chore when there's
more than 2 hosts though, and there is more than 2 hosts needing sync.

## Customized folders, dotfiles and appdata
For some applications, I'd like their dotfiles folder synced into the same location locally. Applications like
OpenCode, Pi, OMP and others have configurations I would like "easily deployable" from another host. This doesn't
necessarily have to be live transferred from another host, some cronjob for selectively backing up important
dotfiles and folders in the user directory.

- Premade folders: Premade folders like "LamaFiles", "dotfiles", "backups" that are available for sync/mount
- Tailscale integration: Find, chose and watch data from host based on a tailscale readout, select hosts for sync/mount etc.
- Dotfile backup: Per-app level. App restore selectable in TUI with "Pull appfiles from: [hosts]".
- Backup: Cronjob with reporting to NTFY or other running services, if running inside docker Dozzle can see logs ie.
- Restore: Restore a previous backup - In original folders with basic preview, or the option to restore into a folder.
- Sync or Mount: Set up a folder to sync or mount, chose from a list of folders ready for sync on tailnet. Needs destination selection.
  - Partial sync/mount: The ability to only mount/sync a folder below the root. Ie. mount `projects` directory inside root share locally.
- Sanity checks: Basic error log checks, faults, permission issues, overwrites etc. Delivers reports to same location backup reports end up.
- Service setup: Set up a systemctl --user to run the filesync service - Managing things like mounts and syncs. Installs prereqs if needed like rclone.
- Centralized storage: Folders CAN be synced

All in all alot of what I want done is supported using SeaFile - I currently have little experience inside SeaFile's
CLI, but I know there's both a "sync" and a "drive" client for shared folders, testing out how well working this is
should be high on the priority list, but what happens when one atleast syncs repo's with tens of thousands of files
is the sync client just choking completely up and needing a restart. Most of this happens during download of an
absolutely insane file list. Mounting folders would avoid all this, but I'm not sure how solid the "working speed"
inside the directories would be? My only thought is using something like rclone mount and setting up some custom
settings to cache files in the mount - It should be supported without breaking the sync.
